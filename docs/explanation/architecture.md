# Architecture

## Overview

The platform follows a **two-phase deployment model**:

1. **Phase 1 -- OpenTofu**: Bootstraps infrastructure and all Kubernetes stacks in strict dependency order. This is the "day-1" path that takes a blank environment to a fully operational platform.

2. **Phase 2 -- Flux v2**: Takes over day-2 reconciliation via GitOps. HelmReleases and Kustomize overlays ensure drift detection and self-healing. After Flux is running, OpenTofu hands off ownership via `tofu state rm`.

## Deployment Pipeline

```
kms-bootstrap (once, podman)
    | --> OpenBao KMS 3-node Raft
    | --> vault-backend :8080 (state storage)
    | --> PKI Root CA + Sub-CAs
    |
env-apply (scaleway/local/outscale)
    | --> kubeconfig --> ~/.kube/talos-$(ENV)
    |
k8s-cni         <-- Cilium MUST be first (~30s)
    |
    +---- parallel (make -j2) ----+
    |                             |
k8s-pki                       k8s-monitoring
    |                             |
openbao-init                      |
    |                             |
k8s-identity                      |
    |                             |
    +---- parallel (make -j2) ----+
    |                             |
k8s-security                  k8s-storage
    |                             |
    +---- flux-bootstrap ---------+
              |
              v
         Flux day-2 (GitOps reconciliation)
         +-- ESO --> OpenBao secrets --> K8s Secrets
         +-- HelmReleases (drift detection, self-healing)
         +-- Kustomize overlays (per-environment)
```

## Why this ordering?

- **Cilium first**: It is the CNI. Without it, no pods can be scheduled on any node.
- **PKI and monitoring parallel**: They have no dependency on each other, only on Cilium.
- **openbao-init after PKI**: The in-cluster OpenBao pods must be running before initialization.
- **Identity after PKI**: Hydra needs TLS certificates from cert-manager's ClusterIssuer.
- **Security and storage parallel**: Both depend on identity (for OIDC/policy context) but not on each other.
- **Flux last**: It needs all stacks present to reconcile them.

## State Storage

All OpenTofu states are stored in OpenBao KV v2, accessed through vault-backend:

```
OpenTofu --HTTP--> vault-backend (:8080) --> OpenBao KV v2 (:8200)
                   +-- /state/scaleway       --> secret/data/tfstate/scaleway
                   +-- /state/k8s-cni        --> secret/data/tfstate/k8s-cni
                   +-- /state/k8s-pki        --> secret/data/tfstate/k8s-pki
                   ...
```

This design avoids cloud-specific state backends (no S3, no GCS). The state backend runs locally in podman, making it usable in air-gapped environments.

- **Authentication**: `TF_HTTP_PASSWORD` env var (token from kms-output/)
- **Encryption**: Raft at-rest (native OpenBao)
- **Locking**: vault-backend creates `-lock` keys in KV v2
- **DR**: `make state-snapshot` creates a Raft snapshot (backup all states at once)

## Secrets Management

Secrets never touch Git. The flow is:

```
OpenBao KV v2 --> ESO ClusterSecretStore --> ExternalSecret --> K8s Secret
```

Application secrets (Hydra, Pomerium, Garage, Harbor) are auto-generated by Terraform `random_id` resources and stored in the Terraform state (which itself lives in OpenBao). ESO refreshes from OpenBao every hour.

## PKI Architecture

A 3-tier CA hierarchy, bootstrapped locally via podman:

```
Root CA (10-year TTL, offline in kms-output/)
    |
    +-- Infra Sub-CA (5-year TTL)
    |       Used by: Cilium mTLS, etcd, internal services
    |       Injected into: cert-manager ClusterIssuer "internal-ca"
    |
    +-- App Sub-CA (5-year TTL)
            Used by: Hydra TLS, user-facing services
            Injected into: cert-manager for app certificates
```

The Root CA private key exists only in `kms-output/` (local, gitignored). Sub-CA certs and keys are injected into Kubernetes secrets by the k8s-pki stack.

## Multi-Environment Strategy

A shared Terraform module (`terraform/modules/talos-cluster`) handles the Talos-specific logic (machine secrets, machine configs, config patches). Each environment calls this module and adds its own cloud resources:

| Environment | Provider | Specifics |
|------------|----------|-----------|
| Scaleway | scaleway/scaleway | 4 stages (IAM, image, cluster, CI), Load Balancer, Private Network |
| Local | libvirt | QEMU/KVM VMs, bridge networking |
| Outscale | outscale | FCU instances |
| VMware | Shell scripts | No Terraform (no vSphere API), OVA with embedded image cache, static IPs |

All K8s stacks are provider-agnostic -- they receive only a `kubeconfig_path` variable and work identically across all environments.

## Stack Boundaries

Each stack owns a clear set of resources and namespaces:

| Stack | Owns | Namespace |
|-------|------|-----------|
| k8s-cni | Cilium | kube-system |
| k8s-monitoring | VictoriaMetrics, VictoriaLogs, Grafana, Headlamp | monitoring |
| k8s-pki | OpenBao x2, cert-manager, CA secrets | secrets |
| k8s-identity | Kratos, Hydra, Pomerium | identity |
| k8s-security | Trivy, Tetragon, Kyverno, Cosign policy | security |
| k8s-storage | local-path, Garage, Velero, Harbor | garage, storage |
| flux-bootstrap | Flux v2, GitRepository, root Kustomization | flux-system |
| external-secrets | ESO, ClusterSecretStore | external-secrets |

## GitOps (Day-2)

After initial deployment, Flux reconciles from `clusters/management/`:

```
clusters/management/
    kustomization.yaml          # Root: references all stacks
    external-secrets/           # ESO + ClusterSecretStore
    k8s-cni/                    # HelmRelease Cilium
    k8s-monitoring/             # HelmReleases monitoring stack
    k8s-pki/                    # HelmReleases PKI stack
    k8s-identity/               # HelmReleases identity stack
    k8s-security/               # HelmReleases security stack
    k8s-storage/                # HelmReleases storage stack
```

Flux watches the Git repository and automatically applies changes. Drift is detected and corrected. HelmReleases are the source of truth for chart versions and values.

## Workload Clusters (CAPI)

The management cluster can provision workload clusters via Cluster API:

- **CAPS** (Scaleway provider): Creates Scaleway instances on demand
- **CAPV** (vSphere provider): Creates VMs with DHCP + MAC reservations (planned)
- **CAPT** (Talos provider): Configures Talos OS on provisioned machines

Workload cluster manifests live in `configs/capi/`.
