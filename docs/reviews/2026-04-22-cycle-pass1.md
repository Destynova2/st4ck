## 1 — tfstate committed to git

**Path:** `envs/scaleway/iam/terraform.tfstate*`
**Why critical:** tfstate contains secret outputs and access keys. Once committed, even after `git rm`, history retention exposes them.
**Fix:**
1. `git rm --cached envs/scaleway/iam/terraform.tfstate envs/scaleway/iam/terraform.tfstate.backup`
2. Add to `.gitignore` at repo root: `**/terraform.tfstate`, `**/terraform.tfstate.*`, `**/.terraform/`
3. Verify the iam stack uses the vault-backend `backend.tf` (consistent with the other stacks under `stacks/*/backend.tf`)
4. Document in `envs/scaleway/iam/README.md` (or top of `main.tf`) that state lives in OpenBao KV v2 like the rest

## 3 — k8s-down.sh `kubectl delete ns --all` without confirmation

**Path:** the audit input cites `scripts/k8s-down.sh:42`. That file is not present at that path in the current repo (`scripts/` only contains `cosign-sign.sh`). The destructive `kubectl delete ns --all` lives in the day-2 teardown chain — likely under one of:
- the `bootstrap-stop` Make target
- a stack-level teardown referenced by the Makefile
**Fix:** wherever found, gate the destructive command behind:
```bash
read -rp "DESTRUCTIVE: delete all namespaces in cluster $(kubectl config current-context)? Type 'YES' to confirm: " ans
[ "$ans" = "YES" ] || { echo "Aborted."; exit 1; }
```
Or accept `--dry-run` / `CONFIRM=YES` env var bypass for CI use.

## 4 — Hardcoded `OPENBAO_TOKEN=root` fallback

**Path:** referenced as `bootstrap/tofu/main.tf:88`. That file does not exist (`bootstrap/tofu/` contains gitea.tf, pki.tf, providers.tf, variables.tf, vault.tf, woodpecker.tf). The fallback is likely in one of these — probably `vault.tf` or `providers.tf`.
**Fix:** replace any `default = "root"` on the OpenBao token variable with:
```hcl
variable "openbao_token" {
  type      = string
  sensitive = true
  validation {
    condition     = length(var.openbao_token) > 8 && var.openbao_token != "root"
    error_message = "OPENBAO_TOKEN must be set to a non-default secret value."
  }
}
```

## 5 — OpenBao seal key plaintext (ADR-014)

**Path:** `stacks/pki/main.tf` plus `stacks/pki/secrets.tf`
**Why critical:** ADR-014 mandates KMS-wrapped seal or SealedSecret. Current code uses a static seal stored in plaintext in the K8s Secret.
**Fix options:**
- A. Wrap with cloud KMS (Scaleway KMS, AWS KMS, etc.) — preferred path per ADR-014
- B. Use Bitnami SealedSecret if KMS not available in dev (sealed at rest in git)
- C. Document the deviation explicitly in ADR-014 with "deferred to next sprint" status

The brigade in `tier3-pass1` will pick option **C** (document the deviation + open a follow-up issue) since options A/B require infra work outside this sprint's scope. The status moves from "drift" to "tracked drift".

## 6 — `start-builder` step missing `path:` filter

**Path:** `.woodpecker.yml` (single file at repo root, NOT `.woodpecker/scaleway-build.yml` as the audit cited — that path doesn't exist; this is a monolithic Woodpecker config). The `start-builder` step lines ~34-54.
**Why critical:** every push to main spins up a Scaleway builder VM even when the change is doc-only — that's billable IaaS time.
**Fix:** add a `path:` filter to the existing `<<: *deploy_only` anchor or inline:
```yaml
when:
  - event: push
    branch: main
    path:
      - "envs/scaleway/image/**"
      - "modules/talos-cluster/**"
      - "vars.mk"
```

## 12 — Hydra `random_id` missing `keepers`

**Path:** `stacks/identity/main.tf` (the `random_id` resource for `hydra_system_secret`)
**Why critical:** without `keepers`, `tofu apply` regenerates the secret on any plan-changing config edit, breaking running Hydra sessions.
**Fix:**
```hcl
resource "random_id" "hydra_system_secret" {
  byte_length = 32
  keepers = {
    namespace = kubernetes_namespace.identity.metadata[0].name
  }
}
```
The keeper ties the secret's lifecycle to the namespace existence, not to provider metadata.
